**ID:** US_04.004 | Multi-configuration Project Configuration > Configure Source Code Management

**Description:**

**As a** user,
**I want to** configure Source Code Management for a Multi-configuration project,
**So that I can** connect the project to my repository and automatically pull the latest code for builds.

**Preconditions**
1. Jenkins instance is up and running and accessible to the user.
2. User is authenticated and has permissions to configure jobs.
3. User has already created a Multi-configuration project and is on the configuration page.
4. User is viewing the **Source Code Management** section of the Multi-configuration project configuration.

**Acceptance Criteria:**
1. In the **Source Code Management** section, the following radio buttons are available:
 - None (selected by default)
 - Git
2. Clicking the **Git** radio button opens the following options:
 - **Repositories**:
     - Repository URL. Specify the Git repository URL (HTTPS, SSH, git:// or a local path).
     - Credentials. Select credentials to access the repository. None (selected by default).  If none selected — anonymous access is used.
    Using the Add button you can choose the following types of credentials:

        - Username with password
        - GitHub App
        - SSH Username with private key
        - Secret file
        - Secret text
         - Certificate 
     -  Advanced dropdown:
        -  Name. Provide a name for the remote repository. If left empty, Jenkins will generate a unique name.
        -  Refspec. Set the refspec to fetch specific refs. Multiple refspecs can be specified separated by spaces. Default is all branches.
     -  **Add Repository** button. Add another repository.
  -  **Branches to build**:
     -  Branch Specifier. Specify branches to track and build. Supported formats: , refs/heads/, /, refs/tags/, , ${ENV_VAR}, wildcards (*, **), regular expressions (:regexp). Default is */master (all repos, master branch).
     -  **Add Branch** button. Add another Branch.
 - **Repository browser**:
     - Choose an external system to browse changes (Auto — automatic detection). Other version control systems are also available:

         - AssemblaWeb
         - FishEye
         - Kiln
         - Microsoft TFS / Azure DevOps
         - bitbucketserver
         - bitbucketweb
         - cgit (CGit)
         - gitblit
         - githubweb
         - gitfiles
         - gitlab
         - gitlist
         - gitoriousweb
         - gitweb
         - gogs
         - phabricator
         - redmineweb
         - rhodecode
         - stash
         - viewgit
 - **Additional Behaviours**:
     - Using the Add button you can choose the following additional behaviours:
         - Advanced checkout behaviours	
        - Advanced clone behaviours	
        - Advanced sub-modules behaviours	
         - Build single revision only	
         - Calculate changelog against a specific branch	
         - Check out to a sub-directory	
         - Check out to specific local branch	
         - Clean after checkout	
         - Clean before checkout	
         - Create a tag for every build	
         - Custom SCM name	
         - Custom user name/e-mail address	
         - Don't trigger a build on commit notifications	
         - First build changelog	
         - Force polling using workspace	
         - Git LFS pull after checkout	
         - Merge before build
         - Polling ignores commits from certain users	
         - Polling ignores commits in certain paths	
         - Polling ignores commits with certain messages	
         - Prune stale remote-tracking branches	
         - Prune stale tags	
         - Sparse Checkout paths	
         - Strategy for choosing what to build	
         - Use commit author in changelog	
         - Wipe out repository & force clone
