**ID:** 02.009 | Freestyle Project Configuration > General project Settings

**Description:**
**As a** user,
**I want to** configure the general project settings for an Item (Item type: Freestyle project),
**So that I can** define basic project properties and build retention rules before setting up build steps.

**Preconditions**
- Jenkins instance is up and running and accessible to the user.
- User is authenticated and has permissions to configure jobs.
- User has already created a Freestyle project and is on the configuration page.
- User is viewing the "General" section of the Freestyle project configuration.

**Acceptance Criteria:**
1. By navigating to the Freestyle project configuration page, the user is presented with the General section (with a gear icon) at the top of the page, which contains the following configuration options — checkboxes:
   
 - **Discard old builds**:
      - Strategy dropdown defaults to Log rotation.
      - Days to keep builds. Set the number of days to keep builds; build records are retained for up to that many days.
      - Max # of builds to keep. Set the Max # of builds to keep; only up to this number of build records are kept.
      - Advanced dropdown:
           - Days to keep artifacts. Set the Days to keep artifacts; artifacts from builds older than this number of days will be deleted, while build logs, history, and reports are retained.
            - Max # of builds to keep with artifacts. Set the Max # of builds to keep with artifacts; only up to this number of builds will have their artifacts retained.
            - Radio button group labeled **Remove last build**. Select **Yes** or **No** for Remove last build to control whether the most recent build is deleted by the retention policy.
  
 - **GitHub project**:
      - Project url. Enter the URL for the GitHub hosted project.
      -  Advanced dropdown:
            -  Display name. Provide a short, clear Context Name for commit status, if left empty, the job name will be used.
  
 - **This project is parameterized** with the ability to add the following parameters:
      -  Boolean parameter:
            -  Name. Enter the parameter name; this name will be used to access the parameter during the build via environment variables.
            -  Set by Default. Check this box to set the parameter's default value.
            -  Description. Add a description that will be shown to the user when starting the build, explaining the parameter's purpose.
      -  Choice parameter:
            -  Name. Specify the parameter name; this name will be used to access the parameter during the build via environment variables.
            -  Choices. Specify the possible values for the parameter, one per line; the value on the first line will be used as the default.
            -  Description. Add a description that will be shown to the user when starting the build, explaining how to choose from the provided list.
      -  Сredentials parameter:
            -  Name. Specify the parameter name; this name will be used to access the credentials identifier (ID) during the build via environment variables.
            -  Credential type. Select the credential type from the list: Any, Username with password, SSH Username with private key, Secret file, Secret text, Certificate — to restrict the available selection options.
            -  Required. Check this box to remove the empty option from the dropdown; choosing credentials will become mandatory (if matching credentials are available).
            -  Default Value. Select the default credentials from those available; they will be preselected when the build is started.
            -  Description. Add a description that will be shown to the user when starting the build, explaining which credentials should be chosen.
      -  File parameter:
            -  File Location. Specify the relative path in the workspace where the uploaded file will be saved.
            -  Description. Add a description that will be shown to the user when starting the build, explaining which file should be uploaded.
            -  `Note: File upload is optional — if the user does not upload a file, the parameter is skipped and existing files in the workspace are not removed. The uploaded file's name is available via an environment variable with the same name as the File Location.`
      -  Multi-line string parameter:
            -  Name. Specify the parameter name; this name will be used to access the parameter value during the build via environment variables.
            -  Default Value. Enter the default value (multi-line text) that will be prefilled in the input field so the user does not have to type it manually for each build.
            -  Description. Add a description that will be shown to the user when starting the build, explaining what multi-line text should be provided. 
      -  Password parameter:
            -  Name. Specify the parameter name; this name will be used to access the password value during the build via an environment variable.
            -  Default Value. Enter the default password value (optional) so the user does not have to type it manually for each build.
            -  Description. Add a description that will be shown to the user when starting the build, explaining what this password is used for.
      -  Run parameter:
            -  Name. Specify the parameter name; this name will be used to access information about the selected build via environment variables during the build.
            -  Project. Specify the project (job) name from which the user can choose builds; the latest build will be offered as the default value.
            -  Filter. Choose a filter to display builds in the dropdown: All Builds (includes running builds), Completed Builds Only, Successful Builds Only (only successful — Stable and Unstable), or Stable Builds Only.
            -  Description. Add a description that will be shown to the user when starting the build, explaining which build should be selected.
      -  String parameter:
            -  Name. Specify the parameter name; this name will be used to access the string value during the build via environment variables.
            -  Default Value. Enter the default value (optional) so the user does not have to type it manually for each build.
            -   Description. Add a description that will be shown to the user when starting the build, explaining what string value should be provided.
            -  Trim the string. Check this box to automatically remove whitespace at the beginning and end of the string entered by the user.
            
 - **Throttle builds**:
      - Number of builds. Specify the maximum number of builds allowed during the specified time period.
      - Time period. Choose the unit of the time period: Second, Minutes, or Hours, during which the limit applies.
      - Allow user triggered builds to skip the rate limit. Check this box to allow builds manually triggered by a user to bypass the rate limit (such builds are still counted toward the limit but are not blocked when the limit is reached).
 - **Execute concurrent builds if necessary**:
      - Execute concurrent builds if necessary. Check this box to allow parallel execution of multiple builds of the same project (by default builds run sequentially). Useful for projects with long test suites or parameterized builds that do not depend on each other.

     `Note about workspace: When concurrent builds are enabled, Jenkins creates a separate workspace for each build by appending a suffix @num to the name (for example, @2). If the Use custom workspace option is enabled, all builds use the same workspace — take care to avoid conflicts from concurrent file access.`



 2. **Advanced contains the following checkboxes**:
 - **Quiet period**: Specify the delay time in seconds before starting the build (default 5 seconds); the delay allows multiple commits to be batched into a single build, reducing queue load and Jenkins strain.
 - **Retry Count**: Specify the number of checkout attempts from the source control management (SCM) system before the build is marked as failed (default 0 — fail on the first error); there is a 10-second delay between attempts. If the option is not checked, the system default from Configure System is used.
 - **Block build when upstream project is building**: Check this box to prevent a build from starting when any upstream (direct or transitive) dependent projects are queued or running. For cyclic dependencies, Jenkins breaks the cycle starting from the oldest requested build.
 - **Block build when downstream project is building**: Check this box to prevent a build from starting when any downstream (direct or transitive) child projects are queued or running. For cyclic dependencies, Jenkins breaks the cycle starting from the oldest requested build.
 - **Use custom workspace**:  Check this box to manually specify the workspace location instead of using the standard directory allocated by Jenkins.
      - Directory — specify the path to the custom workspace (relative paths are allowed, relative to the agent's "remote FS root" or $JENKINS_HOME on the controller). 
 - **Keep the build logs of dependencies**: Check this box to protect from log rotation (automatic deletion) all builds referenced by this project.
      And text field Display Name: Specify the display name of the project that will be shown in the Jenkins web interface instead of the project name.
