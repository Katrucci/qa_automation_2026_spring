**ID:** US_10.014 | Manage Jenkins > Security

**Description:**

**As a** user,
**I want to** configure **Security settings** in the Manage Jenkins section,
**So that I can** define who is allowed to access/use the system.

**Preconditions:**

1. Jenkins instance is up and running and accessible to the user.
2. User is authenticated and has **Administer** permissions.
3. User has navigated to **Manage Jenkins** (via the **Manage Jenkins** link — the gear icon — on the right side of the dashboard).
4. User is viewing the **Security** parameter — shown with a lock icon — in the **Security** section.

**Acceptance Criteria:**

1.  By navigating to the **Security** settings page, I see the following options:
- **Authentication:** I can disable "Keep me signed in" by selecting this checkbox to remove the "Keep me signed in" option from the login screen.
    - **Security Realm:** I can select the authentication source for Jenkins users from the dropdown:
      - Jenkins' own user database (selected by default)
      - Delegate to servlet container
       - GitHub Authentication Plugin
       - LDAP
       - None
  
      - **Allow users to sign up:** I can enable self-registration by selecting this checkbox, allowing users to create their own accounts via the "sign up" link at the top right of the page; when I leave it unchecked, only an administrator can create accounts (default).
  
  
   - **Authorization:**  I can select the authorization strategy for Jenkins from the dropdown: 
       - Logged-in users can do anything (selected by default)
      - Anyone can do anything
       -  GitHub Committer Authorization Strategy(option available only after installing the GitHub Authentication plugin)
       - Legacy mode
       - Matrix-based security
       - Project-based Matrix Authorization Strategy.
      - **Allow anonymous read access:** I can allow unauthenticated users to access Jenkins in a read-only mode by selecting this checkbox.
  
- **Markup Formatter:** I can select the formatter that controls how my descriptions (for jobs, views, builds, etc.) are rendered in Jenkins. I choose from the dropdown:
  - Plain text (selected by default)
  - Safe HTML 


 - **Agents:** I can select the TCP port mode for inbound agents:
      - Fixed 
      - Random 
      - Disable (selected by default)


  - **CSRF Protection:** I can select the **Crumb Issuer**: Default Crumb Issuer is available. I can also **enable Proxy Compatibility** by checking this option if an HTTP proxy sits between my browser client and my Jenkins server, and I receive a 403 response when submitting a form to Jenkins.

 - **Git plugin notifyCommit access tokens:**
    - **Current access tokens:** I can manage access tokens for the notifyCommit endpoint. I can click **Add new access token**, provide an **Access token name**, and click **Generate** to create a new token. These tokens authenticate requests to the notifyCommit endpoint.

 - **Prism syntax highlighting:** I can specify absolute paths of **Permitted Source Code Directories** on agents by clicking Add and entering the **Absolute path of permitted directory**. 

  - **Git Hooks:** 
      - Allow on Controller: I can allow git hooks to run on the Jenkins Controller by checking this option. It is recommended to keep this **disabled** on the controller. 
       - Allow on Agents: I can allow git hooks to run on Jenkins Agents by checking this option. It is recommended to keep this **disabled** on agents as well.

  - **Hidden security warnings:** I can hide security warnings by unchecking individual warning entries in the security warnings list (when warnings are present). Checked warnings (default) are shown to administrators. Unchecked warnings are hidden.

If the list is empty, no security warnings apply to my current Jenkins configuration.

*Note: I cannot hide all warnings for a component at once — only individual warnings can be unchecked. To disable the administrative monitor entirely (hiding all warnings), I can configure that separately in **Configure System**.*

 - **API Token:** 
     - I can enable or disable **Generate a legacy API token for each newly created user** (not recommended — legacy tokens are deprecated).
    - I can enable or disable **Allow users to manually create a legacy API token** (not recommended — legacy tokens are deprecated).
    - I can enable **Enable API Token usage statistics** (selected by default). Recommended.

 - **Content Security Policy:** I can check or uncheck **Enforce Content Security Policy**. When unchecked (disabled), CSP enforcement is turned off. When checked, CSP is enforced.

 
  - **Git Host Key Verification Configuration:** I can select a Host Key Verification Strategy from the dropdown:
      - **Known hosts file (default)** — verifies all host keys using the known_hosts file.
      - **Accept first connection** — automatically adds new host keys to known_hosts, but does not allow connections to previously-seen hosts with modified keys.
      - **Manually provided keys** — verifies all host keys using a manually configured set of keys.
      - **No verification (not recommended)** — does not verify host keys at all.

   - **Sandbox Configuration:** I can check or uncheck **Force the use of the sandbox globally in the system**. When checked, it controls whether the "Use Groovy Sandbox" option is shown to users without Overall/Administer permission.
